\subsection{Slide 7: Frontend Component Architecture}

\subsection{Slide 8: Ingestion: EXIF Coordinate Parsing}

    \item \textbf{Speaker Script:} 
    \textit{``For photo ingestion, I implemented client-side EXIF parsing. When a user selects an image, the frontend reads the file bytes locally using exifr. It extracts GPS coordinates and autofills the form before the upload begins. This removes parsing work from the backend server.''}

\subsection{Slide 9: Advanced EXIF: Yaw, Pitch, Roll \& Altitude}
\begin{itemize}
    \item \textbf{On the slide:} The importance of sensor metadata: altitude, gimbal angles (yaw/pitch/roll) stored in XMP schemas, and their use in calculating photo footprints.
    \item \textbf{Speaker Script:} 
    \textit{``In professional inspections, knowing only the sensor's coordinates isn't enough. We need advanced EXIF telemetry, including sensor altitude and camera gimbal angles—specifically yaw, pitch, and roll. Extracting these allows us to map the exact geographical view and footprint of the photo, rather than rendering a simple coordinate pin.''}
\end{itemize}

\subsection{Slide 10: Smart User Fallbacks: Interactive Mini-Map}
\begin{itemize}
    \item \textbf{On the slide:} The problem of missing EXIF data. The fallback UI: manual coordinate inputs and interactive Leaflet map clicks updating input state immediately.
    \item \textbf{Speaker Script:} 
    \textit{``Since many images have GPS tags stripped due to export settings or privacy rules, I designed an interactive mini-map fallback inside the upload modal. If EXIF data is empty, the user can click on the map, which captures the click event and updates the coordinates instantly.''}
\end{itemize}

\subsection{Slide 11: Map Rendering: Leaflet Marker Clustering}
\begin{itemize}
    \item \textbf{On the slide:} The lag bottleneck of rendering raw DOM markers. The cluster strategy using Supercluster grouping.
    \item \textbf{Speaker Script:} 
    \textit{``Rendering hundreds of individual DOM markers in the browser is slow. I resolved this by wrapping our markers in react-leaflet-cluster. It groups nearby points into single nodes with a count badge. As the user zooms in, it expands them dynamically, keeping map panning at 60 frames per second.''}
\end{itemize}

\subsection{Slide 12: Client-Side Bounding Box Event Handlers}
\begin{itemize}
    \item \textbf{On the slide:} Bounding box formula: Southwest/Northeast bounds. Leaflet `moveend` listener updating bounds state.
    \item \textbf{Speaker Script:} 
    \textit{``To prevent downloading the entire database on load, I set up viewport queries. We listen for Leaflet's moveend events, capture the Southwest and Northeast map corners, and request only the visible photos from the API. We also debounce these events to avoid spamming the backend during rapid panning.''}
\end{itemize}

\subsection{Slide 13: Backend Tech Stack}

    \item \textbf{Speaker Script:} 
    \textit{``Here is my backend stack. I chose Node.js with Express for the API server. We use SQLite/LibSQL for local relational storage, Sharp for image resizing and rotation, and jsonwebtoken for session authentication.''}

\subsection{Slide 14: REST API Endpoint Design}

    \item \textbf{Speaker Script:}

\subsection{Slide 15: Relational Data Model}

    \item \textbf{Speaker Script:} 
    \textit{``I designed a relational database schema. The users table stores logins. The photos table links to users via a foreign key, and comments link to both photos and users. I configured cascade deletion rules, so deleting a photo automatically cleans up its associated comment threads.''}

\subsection{Slide 16: Relational Indexing Strategy}

\subsection{Slide 17: Media Pipeline: Normalization \& Compression}
\begin{itemize}
    \item \textbf{On the slide:} The EXIF orientation bug explained. Sharp code pipeline: `sharp().rotate().jpeg({ quality: 80 }).toFile()`.
    \item \textbf{Speaker Script:} 
    \textit{``Mobile photos often contain orientation flags that cause browsers to render them rotated. During upload, my backend interceptor routes the image buffer to Sharp, calls the rotate method to permanently fix the pixels, and compresses the image to JPEG at 80 percent quality to save storage space.''}

\subsection{Slide 18: Asynchronous Processing Pipeline}
\begin{itemize}
    \item \textbf{On the slide:} Diagram showing how the upload response completes before background AI tasks are launched.
    \item \textbf{Speaker Script:} 
    \textit{``Calling AI APIs takes time. I designed the server to save the coordinates and immediately return a HTTP 201 response. Once the user's connection is closed, the backend runs the AI generator in the background. This ensures that uploading photos remains near-instant.''}
\end{itemize}

\subsection{Slide 19: Asynchronous AI: Vision Model Integration}

    \item \textbf{Speaker Script:}

\subsection{Slide 20: AI Resilience: Quota Handling \& Exponential Backoff}
\begin{itemize}
    \item \textbf{On the slide:} Description of the backoff retry algorithm (maximum 3 attempts, doubling sleep delays: 2s $\to$ 4s $\to$ 8s).
    \item \textbf{Speaker Script:} 
    \textit{``Third-party APIs face rate limits and network drops. If the Gemini API call fails, my code executes an exponential backoff loop. It retries up to 3 times, doubling the delay on each failure. If all attempts fail, it logs the error and leaves the description empty, preventing backend crashes.''}
\end{itemize}

\subsection{Slide 21: Dynamic Regeneration: On-Demand AI Updates}

    \item \textbf{On the slide:} Manual AI regeneration flow via `POST /photos/:id/regenerate-description`.
    \item \textbf{Speaker Script:} 
    \textit{``If the AI fails during upload or is disabled initially, users can click 'Regenerate' in the detail modal. This sends a request to our regenerate endpoint, runs the AI worker, updates the database, and returns the new text to update the UI dynamically.''}
\end{itemize}

\subsection{Slide 22: Stateless Authentication: JWT Verification}
\begin{itemize}
    \item \textbf{On the slide:} Stateless authentication diagram. The process of verification using JWT signatures.
    \item \textbf{Speaker Script:} 
    \textit{``We use stateless JWT authentication. The server doesn't store session IDs. It signs a token on login using a secure key. The client attaches this token to HTTP headers, and our middleware decodes it to verify the user's session instantly.''}
\end{itemize}

\subsection{Slide 23: Cryptographically Secure OTP Generation}
\begin{itemize}
    \item \textbf{On the slide:} Security vulnerability of Math.random. Secure code generation using `crypto.randomInt()`.
    \item \textbf{Speaker Script:} 
    \textit{``The original code used Math.random to create 8-digit verification codes. Since Math.random follows predictable patterns, I refactored the auth router to use Node's native crypto library. This generates cryptographically secure, unpredictable numbers using hardware entropy.''}
\end{itemize}

\subsection{Slide 24: Code Lifecycle Security: TTL \& Single-Use Cleanup}
\begin{itemize}
    \item \textbf{On the slide:} Code lifecycle settings: 10-minute expiry time-to-live (TTL) and instant map deletion upon validation.
    \item \textbf{Speaker Script:} 
    \textit{``To protect the authentication flow, I implemented token lifecycle rules. Verification codes are saved in memory with a strict 10-minute TTL. Expired codes are rejected, and valid codes are deleted immediately on use to block brute-force and replay attacks.''}
\end{itemize}

\subsection{Slide 25: Code Quality Refactoring: Clean Architecture}
\begin{itemize}
    \item \textbf{On the slide:} Refactoring changes: separating routing, removing unused frontend states, and cleaning mailer debug texts.

    \textit{``I audited and refactored the codebase for quality. I separated routes and middleware, removed dead variables like verificationCode from the frontend state, and cleaned out informal debug messages to ensure a professional code standard.''}
\end{itemize}

\subsection{Slide 26: Environment Portability: Configurable APIs}

\subsection{Slide 32: Production Scaling: SQLite Limitations}
\begin{itemize}
    \item \textbf{On the slide:} SQLite limits: file locking on writes, single-process file, lack of native GIS extensions.
    \item \textbf{Speaker Script:} 
    \textit{``As we scale to thousands of photo uploads, SQLite's limitations appear. It locks the entire database file during writes, and since it is local, we cannot scale the backend horizontally. To scale this, our production path requires migrating to PostgreSQL.''}
\end{itemize}

\subsection{Slide 33: Database Migration: PostgreSQL with PostGIS}
\begin{itemize}
    \item \textbf{On the slide:} Storing coordinates in `geom GEOGRAPHY(Point, 4326)` columns. Synchronizing geom data with lat/lng triggers.
    \item \textbf{Speaker Script:} 
    \textit{``In PostgreSQL, we enable the PostGIS spatial extension. We migrate separate latitude and longitude columns into a single geography geometry column using SRID 4326. We configure triggers to update this geometry column automatically whenever coordinate fields change.''}
\end{itemize}

\subsection{Slide 34: Spatial Indexing: GIST on Coordinate Points}
\begin{itemize}
    \item \textbf{On the slide:} SQL command: `CREATE INDEX ... USING GIST(geom)`. Explanation of GIST bounding box hierarchies.
    \item \textbf{Speaker Script:} 
    \textit{``B-Tree indexes cannot handle multi-dimensional map queries. PostGIS uses GIST indexing, which groups coordinate points into bounding box hierarchies. This allows the database to locate photos inside visible regions instantly, avoiding full-table scans.''}
\end{itemize}

\subsection{Slide 35: Bounding Box Queries: SQL ST\_MakeEnvelope}
\begin{itemize}
    \item \textbf{On the slide:} SQL spatial query block using `ST_MakeEnvelope` and the geography overlap operator `&&`.
    \item \textbf{Speaker Script:} 
    \textit{``In production, our bounding box API queries map bounds directly to PostGIS envelopes. By using the ST\_MakeEnvelope function and the double ampersand operator, we query intersections quickly, returning visible points in sub-millisecond times.''}
\end{itemize}

\subsection{Slide 36: Proximity Queries on Linear Assets}
\begin{itemize}
    \item \textbf{On the slide:} SQL proximity query block utilizing `ST_DWithin` to find photos within 50 meters of pipeline segments.
    \item \textbf{Speaker Script:} 
    \textit{``For infrastructure inspections, we can run spatial proximity queries. Using PostGIS ST\_DWithin, we can query all photo coordinates captured within 50 meters of a specific pipeline segment. This automates the correlation of inspection data to grid assets.''}
\end{itemize}

\subsection{Slide 37: Cloud Storage Architecture: S3 \& CDN Delivery}
\begin{itemize}
    \item \textbf{On the slide:} S3 Presigned URL workflow diagram. Serving images through CDN edge caches.
    \item \textbf{Speaker Script:} 
    \textit{``To handle heavy image uploads, we offload media from the API server. The client requests a secure presigned upload URL from the API and uploads files directly to an S3 bucket. Images are then served through CDN caches closer to users to minimize latency.''}
\end{itemize}

\subsection{Slide 38: Retrospective: JSDOM Map Mocking}
\begin{itemize}
