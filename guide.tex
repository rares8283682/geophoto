\section{Core Frontend Techniques \& Implementation}

\subsection{1. Client-Side EXIF Metadata Extraction}
\begin{itemize}
    \item \textbf{What it is:} Automatically reading coordinate metadata embedded directly in image files (JPEG/TIFF) inside the browser before the upload occurs.
    \item \textbf{Implementation Details:}
      \begin{itemize}
        \item Located in: [UploadModal.jsx](file:///Users/raresolteanu/Desktop/HyLight-Tehnical%20test/geophotos/frontend/src/components/Upload/UploadModal.jsx).
        \item Library: We import \texttt{exifr} to process the selected raw file object.
        \item Workflow:
\begin{lstlisting}[language=JavaScript]
// Reads raw file, parses GPS headers, and returns decimals
const gps = await exifr.gps(file);
if (gps && gps.latitude && gps.longitude) {
  setLatitude(gps.latitude.toString());
  setLongitude(gps.longitude.toString());
}
\end{lstlisting}
      \end{itemize}
    \item \textbf{Why it is a best practice:}
      \begin{itemize}
        \item \textbf{UX:} Saves users from manually typing coordinates for photos taken on mobile devices or GPS-enabled inspection equipment (like HyLight's airships).
        \item \textbf{Efficiency:} Ingestion occurs on the client's thread. The server does not waste CPU cycles parsing image byte arrays for metadata.
      \end{itemize}
\end{itemize}

\subsection{2. Leaflet Map Clustering \& Custom Markers}
\begin{itemize}
    \item \textbf{What it is:} Grouping nearby map markers together into aggregated clusters on the map viewport, and rendering customized HTML markers using thumbnail images.
    \item \textbf{Implementation Details:}
      \begin{itemize}
        \item Located in: [MapView.jsx](file:///Users/raresolteanu/Desktop/HyLight-Tehnical%20test/geophotos/frontend/src/components/Map/MapView.jsx).
        \item Libraries: We wrap Leaflet markers in \texttt{MarkerClusterGroup} (utilizing the Supercluster grouping algorithm).
        \item Custom Icons: We construct Leaflet \texttt{L.divIcon} objects to render custom HTML elements containing the photo's S3/Local thumbnail URL inside a round CSS class with glowing selection borders:
\begin{lstlisting}[language=JavaScript]
function makePhotoIcon(photo, isSelected) {
  const imgSrc = `${API_BASE_URL}/uploads/${photo.filename}`;
  return L.divIcon({
    html: `<div class="photo-marker-wrap ${isSelected ? 'selected' : ''}">
             <img src="${imgSrc}" />
           </div>`,
    className: 'custom-photo-icon'
  });
}
\end{lstlisting}
      \end{itemize}
    \item \textbf{Why it is a best practice:}
      \begin{itemize}
        \item \textbf{Performance:} Standard Leaflet markers render as full DOM elements. Rendering thousands of separate DOM nodes causes client-side rendering threads to choke. Clustering reduces active DOM count to a fraction.
        \item \textbf{Aesthetics:} Gives a clean, visual representation of images directly on the map interface.
      \end{itemize}
\end{itemize}

\subsection{3. Viewport-Based Bounding Box Queries}
\begin{itemize}
    \item \textbf{What it is:} Restricting map queries to load only the photo coordinates currently visible inside the user's active browser screen.
    \item \textbf{Implementation Details:}
      \begin{itemize}
        \item Located in: [MapView.jsx](file:///Users/raresolteanu/Desktop/HyLight-Tehnical%20test/geophotos/frontend/src/components/Map/MapView.jsx) \& [MapPage.jsx](file:///Users/raresolteanu/Desktop/HyLight-Tehnical%20test/geophotos/frontend/src/components/Map/MapPage.jsx).
        \item Leaflet Event Listeners:
\begin{lstlisting}[language=JavaScript]
useMapEvents({
  moveend() {
    const bounds = map.getBounds();
    const bbox = {
      south: bounds.getSouth(),
      west: bounds.getWest(),
      north: bounds.getNorth(),
      east: bounds.getEast()
    };
    onBoundsChange(bbox); // Passes viewport coordinates back to MapPage state
  }
});
\end{lstlisting}
        \item The bounds are debounced, and a request is made to:
        \lstinline[language=HTTP]{GET /photos?bbox=south,west,north,east}
      \end{itemize}
    \item \textbf{Why it is a best practice:}
      \begin{itemize}
        \item Prevents the application from downloading the entire global database of photos on load. If a user is viewing Paris, we only fetch photos in Paris.
      \end{itemize}
\end{itemize}

\section{Core Backend Techniques \& Implementation}

\subsection{1. Asynchronous AI Generation Pipeline with Retry Logic}
\begin{itemize}
    \item \textbf{What it is:} Offloading Gemini AI description generation tasks from the main HTTP upload request-response cycle and managing third-party API errors using retry loops.
    \item \textbf{Implementation Details:}
      \begin{itemize}
        \item Located in: [routes/photos.js](file:///Users/raresolteanu/Desktop/HyLight-Tehnical%20test/geophotos/backend/routes/photos.js) \& [regenerate_descriptions.js](file:///Users/raresolteanu/Desktop/HyLight-Tehnical%20test/geophotos/backend/regenerate_descriptions.js).
        \item Asynchronous Trigger:
\begin{lstlisting}[language=JavaScript]
// Express router saves metadata first
const photoId = await db.insertPhoto(metadata);
res.status(201).json({ id: photoId, message: 'Photo uploaded successfully!' });

// Kicks off the AI processing in the background (Non-blocking)
generateAIDescriptionBackground(photoId, filePath);
\end{lstlisting}
        \item Quota & Retries (with Exponential Backoff):
\begin{lstlisting}[language=JavaScript]
async function generateWithRetry(filePath) {
  let attempt = 0;
  while (attempt < 3) {
    try {
      return await callGeminiAI(filePath);
    } catch (err) {
      attempt++;
      if (attempt >= 3) throw err;
      const delay = 2000 * Math.pow(2, attempt); // 2s -> 4s -> 8s
      await sleep(delay);
    }
  }
}
\end{lstlisting}
      \end{itemize}
    \item \textbf{Why it is a best practice:}
      \begin{itemize}
        \item \textbf{Non-blocking UI:} The user does not see a loading spinner for 8 seconds while waiting for the AI response. Uploads complete in milliseconds.
        \item \textbf{Resilience:} Catching quota errors and retrying protects the system from transient rate limits (429) or third-party outages.
      \end{itemize}
\end{itemize}

\section{Security, Authentication \& Code Cleanliness}

\subsection{1. Cryptographically Secure OTP Generation}
\begin{itemize}
    \item \textbf{What it is:} Replacing standard pseudo-random functions with secure, unpredictable number generators.
    \item \textbf{Implementation Details:}
      \begin{itemize}
        \item Located in: [routes/auth.js](file:///Users/raresolteanu/Desktop/HyLight-Tehnical%20test/geophotos/backend/routes/auth.js).
        \item Module: Node's native \texttt{crypto}.
        \item Code Upgrade:
\begin{lstlisting}[language=JavaScript]
// OLD (Vulnerable): Math.floor(10000000 + Math.random() * 90000000)
// NEW (Secure):
const code = crypto.randomInt(10000000, 100000000).toString();
\end{lstlisting}
      \end{itemize}
    \item \textbf{Why it is a best practice:}
      \begin{itemize}
        \item \texttt{Math.random()} is not secure. Its values are generated by a math formula (LGC) that can be predicted if an attacker analyzes a few past outputs. \texttt{crypto.randomInt()} uses hardware entropy sources, making codes unpredictable.
      \end{itemize}
\end{itemize}

\subsection{2. Token Lifecycle Management: Expirations (TTL) \& Single-Use Cleanup}
\begin{itemize}
    \item \textbf{What it is:} Setting strict lifetimes for validation tokens and ensuring they are instantly deleted after use.
    \item \textbf{Implementation Details:}
      \begin{itemize}
        \item Located in: [routes/auth.js](file:///Users/raresolteanu/Desktop/HyLight-Tehnical%20test/geophotos/backend/routes/auth.js).
        \item Code Workflow:
\begin{lstlisting}[language=JavaScript]
// Store code with 10-minute expiry
signupCodes.set(email.toLowerCase(), {
  code,
  expiresAt: Date.now() + 10 * 60 * 1000 // 10-minute TTL
});

// Validation check
const record = signupCodes.get(email.toLowerCase());
if (!record || record.expiresAt < Date.now()) {
  signupCodes.delete(email.toLowerCase()); // Clear expired code
  return res.status(400).json({ error: 'Expired code' });
}

// Clear immediately upon successful login/verification
signupCodes.delete(email.toLowerCase());
\end{lstlisting}
      \end{itemize}
    \item \textbf{Why it is a best practice:}
      \begin{itemize}
        \item \textbf{Prevents Brute-Force:} Expiring tokens prevents hackers from trying every combination over hours.
        \item \textbf{Prevents Replay Attacks:} Once a code is validated, it is deleted. It cannot be used a second time.
      \end{itemize}
\end{itemize}

\subsection{3. Refactoring Unused Code and Hardcoded Secrets}
\begin{itemize}
    \item \textbf{What it is:} Cleaning up dead code, removing console noise, and isolating configuration details from the code.
    \item \textbf{Implementation Details:}
      \begin{itemize}

    \item \textbf{Why it is a best practice:} Shows professionalism. Leaving debug strings or dead states in code looks sloppy to reviewers.
\end{itemize}

\section{Production Scaling Blueprint (For HyLight)}

\subsection{1. PostgreSQL & PostGIS Spatial Extension}
\begin{itemize}
    \item \textbf{Why SQLite is limited:} SQLite is a single file. It does not support concurrent write operations and lacks native geospatial optimization tools.
    \item \textbf{PostGIS Implementation:}
      \begin{itemize}
        \item We store coordinates inside a \texttt{GEOGRAPHY(Point, 4326)} column called `geom` (using standard GPS SRID 4326).
        \item Index coordinates using GIST (Generalized Search Tree):
        \lstinline[language=SQL]{CREATE INDEX idx_geom ON photos USING GIST(geom);}
      \end{itemize}
\end{itemize}

\subsection{2. Viewport & Proximity Queries}
\begin{itemize}
    \item \textbf{Viewport queries:} Filter coordinates using spatial bounding envelopes:
\begin{lstlisting}[language=SQL]
SELECT id, filename FROM photos 
WHERE geom && ST_MakeEnvelope(west, south, east, north, 4326);
\end{lstlisting}
    \item \textbf{Proximity to pipeline corridors:} Find all photos captured within 50 meters of a pipeline segment:
\begin{lstlisting}[language=SQL]
SELECT p.id, p.filename FROM photos p, pipeline_routes l
WHERE l.id = :lineId AND ST_DWithin(p.geom, l.geom, 50.0);
\end{lstlisting}
\end{itemize}

\subsection{3. S3 Presigned Uploads & CDN}
\begin{itemize}
    \item \textbf{S3 Presigned URLs:}
      \begin{itemize}
        \item Frontend requests an upload url: \texttt{GET /get-presigned-url}.
        \item API generates a secure URL using AWS SDK.
        \item Client uploads directly to S3 bucket. Node server resources are untouched.
      \end{itemize}
    \item \textbf{CDN Caching:} Serving images through a CDN (Cloudflare/CloudFront) caches images at regional edges, reducing download times.
\end{itemize}

\section{Retrospective, Challenges \& Key Learnings}
\begin{itemize}
    \item \textbf{Watcher Limits:} Local file reload systems (`npm run dev`) crashed on macOS because the system watch handles reached maximum capacity.
    \item \textbf{Testing Maps in Headless DOMs:} Leaflet maps depend on active browser screen sizes. Simulating this in Node's JSDOM console required writing deep mocks for Leaflet container sizes, teaching me that **Playwright E2E tests** are far better suited for maps than unit tests.
    \item \textbf{Computer Vision Roadmap:} For the future, I want to add YOLO/Segment Anything object detection models into the upload flow to automatically flag anomalies (e.g. rust or cracks on assets) based on coordinate mapping.
\end{itemize}
