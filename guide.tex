\section{Five Core Tech Concepts Explained with Simple Metaphors}
Use these metaphors to explain the tech during the interview. It makes you sound very smart and makes the concepts impossible to forget.

\subsection{1. JWT (JSON Web Token) $\to$ The Festival Wristband}
\begin{itemize}
    \item \textbf{The Old Way (Stateful):} Every time you want to do something, the server checks a giant list (session database) to see if you are logged in. This slows down when there are thousands of users.
    \item \textbf{The JWT Way (Stateless):} When you log in, the server signs a secure ticket (the JWT) and gives it to you. You store it in your browser. Every time you ask for photos or add comments, you show this token.
    \item \textbf{The Metaphor:} It is like a \textbf{Festival Wristband}. When you enter the gate, they check your ID once and give you a wristband. You don't show your ID every time you buy a drink; you just show the wristband. The bartender doesn't check a database; they just look at the wristband's stamp to see if it is real.
\end{itemize}

\subsection{2. CSRNG (Secure Random Numbers) $\to$ Rolling Real Dice vs. Computer Patterns}
\begin{itemize}
    \item \textbf{The Old Way (\texttt{Math.random()}):} The verification code was generated using a basic formula. Computers can't naturally be random; they follow patterns. If a hacker knows the formula, they can guess the next verification code.
    \item \textbf{The CSRNG Way (\texttt{crypto.randomInt()}):} We use Node's secure random number generator. It uses physical measurements from your computer (like temperature or key press timing) to create a truly unpredictable code.
    \item \textbf{The Metaphor:} \texttt{Math.random()} is like a magician pretending to pick a card, but he actually has a pre-determined order. \texttt{crypto} is like throwing a real pair of dice in the air—no one can predict the outcome.
\end{itemize}

\subsection{3. Asynchronous AI $\to$ The McDonald's Ordering Counter}
\begin{itemize}
    \item \textbf{The Problem:} Creating an AI description from a photo takes 5--10 seconds because we have to send it to Google Gemini and wait.
    \item \textbf{The Solution:} We don't make the user wait. The backend saves the coordinates instantly, tells the user ``Done!'' (HTTP 201), and kicks off the AI process in the background. Once the description is ready, it updates the database.
    \item \textbf{The Metaphor:} It's like \textbf{ordering food at McDonald's}. The cashier doesn't make you stand at the cash register for 10 minutes while they cook your fries. They give you a receipt with a number and tell you to wait. You go sit down, and they call your number when the food is ready.
\end{itemize}

\subsection{4. MSW (Mock Service Worker) $\to$ The Hollywood Stunt Double}
\begin{itemize}
    \item \textbf{The Problem:} We want to test our React components without running our real database or backend. Mocking Axios functions directly makes tests fragile.
    \item \textbf{The Solution:} We intercept network requests. The frontend sends HTTP requests exactly like in production, but MSW intercepts them at the browser level and feeds them fake database data.
    \item \textbf{The Metaphor:} It is like a \textbf{Stunt Double} in a movie. The main actor (your React code) thinks they are jumping off a real building (the backend), but the stunt double (MSW) catches them safely. If you change the camera (Axios $\to$ Fetch), the actor's jump still works because the stunt double is still catching them.
\end{itemize}

\subsection{5. PostGIS (Spatial Database) $\to$ Mailboxes organized by Neighborhood}
\begin{itemize}
    \item \textbf{The Problem:} Standard databases search for text or numbers. Searching for points on a 2D map using traditional methods means checking every single photo in the database to see if it is inside your view.
    \item \textbf{The PostGIS Solution:} We use a special database extension (PostGIS) that groups coordinates into spatial grids (GIST indexes).
    \item \textbf{The Metaphor:} Imagine a postman delivering letters. SQLite is like having one giant pile of letters in alphabetical order; to find a letter for a specific street, you have to read through the whole stack. PostGIS is like having \textbf{boxes sorted by neighborhood}. You go directly to the correct neighborhood box and ignore the rest.
\end{itemize}

\section{Anticipated Questions & Direct Answers}
These are the questions Jules will likely ask. Memorize these direct answers.

\subsection{Q1: Why did you use SQLite and how would you scale it?}
\begin{itemize}
    \item \textbf{Your Answer:} 
    ``I used SQLite for the test because it is simple to run locally without setup friction. However, SQLite locks the database file during writes, which bottlenecks concurrent uploads. For production, I would migrate to PostgreSQL and PostGIS. This allows us to handle heavy concurrent writes and run fast map-coordinate queries.''
\end{itemize}

\subsection{Q2: What did you change in the verification codes and why?}
\begin{itemize}
    \item \textbf{Your Answer:} 
    ``The original code used Math.random() which follows computer patterns and can be guessed. I changed it to Node's secure crypto library to make codes unpredictable. I also added a 10-minute expiry and deleted codes immediately after use to prevent replay attacks.''
\end{itemize}

\subsection{Q3: Why is the Gemini AI description generation asynchronous?}
\begin{itemize}
    \item \textbf{Your Answer:} 
    ``AI calls take several seconds. If we wait for the AI to respond before finishing the upload, the user has to wait on a loading screen. Making it asynchronous means the user gets an instant 'Success' response, and the description loads in the background.''
\end{itemize}

\subsection{Q4: What was the hardest part of the tests?}
\begin{itemize}
    \item \textbf{Your Answer:} 
    ``Testing Leaflet maps in a console environment. Leaflet needs a real browser screen size to render, which JSDOM does not have. I had to mock Leaflet event listeners to pass the tests. This taught me that map components are better tested using Playwright E2E tools than simple unit tests.''
\end{itemize}

\subsection{Q5: If an airship drone uploads 50,000 photos, how do we prevent a backend crash?}
\begin{itemize}
    \item \textbf{Your Answer:} 
    ``We should never upload files directly through our Node.js API server. Instead, we use S3 Presigned URLs. The client uploads the images directly to S3. Then, S3 triggers background workers (using a queue like BullMQ) to resize the images, extract GPS data, and run AI detection in the background.''
\end{itemize}

\subsection{Q6: What is MSW and why did you use it?}
\begin{itemize}
    \item \textbf{Your Answer:} 
    ``MSW intercepts network requests at the browser level. Instead of mocking Axios functions directly (which breaks if we switch to Fetch), we test our components using real HTTP requests intercepted by MSW. This validates the actual API contracts.''
\end{itemize}
